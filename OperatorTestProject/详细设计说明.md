# 详细设计说明

## 1. 文档目标

本文档用于说明当前仓库的职责边界、运行流程、模块设计、数据流转、LLM 调用链路，以及后续可改进方向。本文档基于当前代码实现整理，不假设仓库之外的部署平台或外部系统。

仓库目标可以概括为：

1. 输入一份 CANN 算子说明文档（Markdown）。
2. 按预定义模块从文档中提取结构化规则。
3. 对提取结果做 JSON 结构校验和业务校验。
4. 合并各模块输出，生成统一的算子规则 JSON。
5. 基于规则 JSON 自动生成测试用例。

## 2. 总体架构

当前系统是一个以 `src/` 为核心包的 Python CLI 工具，主要由以下部分组成：

- 入口层：负责命令行参数读取和程序启动。
- 配置层：负责读取 `config/config.yaml` 并构建运行参数。
- 工作流层：使用 LangGraph 编排顶层算子处理流程和单模块处理流程。
- 提示词层：`prompts/` 目录中的 Markdown 文件定义了提取与校验规则。
- LLM 接入层：`LLMService` 封装模型实例池、轮询和重试。
- 结果持久化层：负责保存模块提取结果、错误信息、合并结果和测试用例。
- 测试用例生成层：根据最终规则 JSON 生成随机化测试样本。

从职责上看，当前系统的真正主链路是：

`entry_point.py` -> `src.main` -> `src.graph.OperatorProcessingGraph` -> `src.module_processor.ModuleProcessor` -> `src.module_processing_graph.ModuleProcessingGraph` -> `ExtractionNode / JsonValidationNode / ValidationNode` -> `SkillExecutor`

## 4. 核心执行流程

## 4.1 启动流程

### 4.1.1 CLI 入口

`entry_point.py` 从命令行读取算子文档路径，并执行：

- `asyncio.run(main(operator_doc_path))`

该文件的作用主要是打包时保证 `src` 包可正确导入。

### 4.1.2 主程序初始化

`src/main.py` 中的主流程为：

1. 加载配置。
2. 初始化日志。
3. 初始化 `LLMService`、`SkillExecutor`、`ModuleProcessor`。
4. 创建顶层工作流 `OperatorProcessingGraph`。
5. 读取输入文档内容。
6. 执行工作流。
7. 输出最终合并结果路径与测试用例路径。

### 4.1.3 顶层状态

顶层状态定义在 `src/models.py`，主要字段包括：

- `operator_name`
- `operator_doc`
- `module_results`
- `merged_result_path`
- `test_cases_path`
- `current_iteration`
- `error`

其中 `current_iteration` 在当前顶层工作流中基本没有实际作用，属于遗留字段。

## 4.2 顶层工作流设计

`src/graph.py` 使用 LangGraph 定义顶层处理图：

1. `initialize`
2. `extract_modules`
3. `merge_results`
4. `generate_test_cases`
5. `END`

流程是严格串联的，不存在条件分支。

### 4.2.1 initialize

初始化顶层状态，清空历史结果和错误信息。

### 4.2.2 extract_modules

根据配置 `module_execution.mode` 选择：

- 顺序处理 `sequential`
- 并发处理 `concurrent`

无论采用哪种模式，实际都会遍历 `ModuleProcessor.MODULES`：

- `parameter_constraints`
- `basic_info`
- `inter_parameter_constraints`
- `platform_specifics`
- `other_parameters`
- `functions`
- `dtype_map`

### 4.2.3 merge_results

如果任何模块处理失败，顶层状态会写入 `error`，随后跳过合并。

如果全部成功，则收集每个模块的 `result_path`，交给 `SkillExecutor.merge_operator_rules()` 做字典合并，并将结果写到：

- `workspace/<operator>/<operator>_extracted_constraints.json`

### 4.2.4 generate_test_cases

基于合并后的规则文件调用 `test_case_generator.generate_test_cases_from_file()` 生成测试用例，输出为：

- `workspace/<operator>/<operator>_cases.json`

## 5. 单模块工作流设计

## 5.1 设计目标

单模块工作流负责把“文档中的一个模块”提取为结构化 JSON，并做两级校验：

1. JSON 结构正确
2. 业务内容正确

其核心思路是“双层迭代”：

- 小循环：`extract <-> json_validate`
- 大循环：`extract <-> validate`

## 5.2 状态模型

`src/module_graph_state.py` 中 `ModuleProcessingState` 包含：

- 基本输入
  - `module_name`
  - `operator_name`
  - `operator_doc`
- 双层迭代控制
  - `outer_iteration`
  - `max_outer_iterations`
  - `inner_iteration`
  - `max_inner_iterations`
- 中间结果
  - `extracted_data`
  - `extraction_result`
  - `version_dir`
  - `previous_error_path`
  - `json_validation_success`
  - `json_validation_error`
  - `validation_result`
  - `validation_error`
- 最终结果
  - `final_result_path`
  - `final_error_path`
  - `total_iterations`
- 全局错误
  - `error`

## 5.3 单模块状态机

`src/module_processing_graph.py` 定义的节点：

1. `extract`
2. `json_validate`
3. `validate`
4. `success`
5. `fail`

### 5.3.1 extract

由 `ExtractionNode` 执行，职责：

1. 创建模块版本目录，如 `workspace/<operator>/<module>/v0`
2. 加载提取规则文件 `prompts/<module>_extracting_guide.md`
3. 构建提示词
4. 调用 LLM
5. 解析 JSON
6. 保存模块结果 `<module>.json`

### 5.3.2 json_validate

由 `JsonValidationNode` 执行，职责：

1. 将提取结果转回 JSON 字符串。
2. 使用 `json_to_model_loader.load_json_to_model()` 与 Pydantic 模型对照校验。
3. 如果失败，将详细错误写回状态中的 `json_validation_error`。
4. 增加 `inner_iteration`，并回到 `extract` 重试。

### 5.3.3 validate

由 `ValidationNode` 执行，职责：

1. 加载校验规则文件 `prompts/<module>_check_guide.md`
2. 读取刚生成的模块 JSON
3. 构建“业务校验提示词”
4. 调用 LLM 输出 `CheckError[]`
5. 若非空数组，则保存 `error.md`
6. 将错误信息回写到状态中的 `validation_error`
7. 增加 `outer_iteration`，重新进入 `extract`

### 5.3.4 success/fail

- `success`：记录最终结果路径并计算迭代次数。
- `fail`：记录最终错误路径并结束。

## 6. 提示词体系设计

## 6.1 规则文件组织方式

每个模块使用两类提示词：

- 提取提示词：`<module>_extracting_guide.md`
- 校验提示词：`<module>_check_guide.md`

好处：

- 模块职责清晰
- 提取与校验规则分离
- 规则可独立迭代

代价：

- 规则文件数量多
- 规则和代码模型容易漂移

## 6.2 PromptBuilder 设计

`PromptBuilder` 不存业务规则本身，只负责拼装：

- 原始文档
- 规则内容
- JSON 校验错误反馈
- 业务校验错误反馈

因此它相当于“通用提示词外壳”，而不是“规则来源”。

## 6.3 错误反馈闭环

当前系统建立了两个反馈通道：

1. `json_validation_error`
   - 由 `JsonValidationNode` 生成，回灌到下一轮提取提示词
2. `validation_error`
   - 由 `ValidationNode` 生成，回灌到下一轮提取提示词

这使系统形成“提取 -> 发现错误 -> 再提取”的闭环。

## 7. 数据模型设计

## 7.1 顶层规则模型

`common_model_definition.py` 定义了最终目标 JSON 结构，包括：

- `OperatorRule`
- `OperatorFunction`
- `Parameter`
- `ParamConstraints`
- `OtherParameters`
- `DtypeMapItem`
- `InterParamConstraint`
- `PlatformSpecific`

这些模型既是：

- JSON 结构校验规范
- 提示词中的目标 schema 参考
- 测试用例生成器的输入契约

## 7.2 校验错误模型

`check_error.py` 中定义：

- `error_path`
- `error_message`
- `fix_suggestion`
- `is_fixed`

业务校验期望 LLM 返回该结构的数组。

## 8. 持久化与输出设计

## 8.1 模块级中间产物

每个模块会在 `workspace/` 下生成版本目录，形如：

```text
workspace/
  <operator>/
    <module>/
      v0/
        <module>.json
        error.md
      v1/
        <module>.json
        error.md
```

这使得每轮提取与校验结果都能被追踪。

## 8.2 最终输出

最终系统会生成两个核心文件：

- `<operator>_extracted_constraints.json`
- `<operator>_cases.json`

前者是最终规则，后者是测试用例。

## 9. LLM 调用链路设计

## 9.1 当前调用位置

当前仓库只有两条真正的 LLM 业务调用链：

### 9.1.1 提取链路

`ExtractionNode._call_llm()` -> `LLMService.invoke_with_retry()` -> `LLMService.invoke()` -> `llm.ainvoke(messages)`

用途：

- 根据文档和提取规则生成结构化 JSON。

### 9.1.2 业务校验链路

`ValidationNode._call_llm()` -> `LLMService.invoke_with_retry()` -> `LLMService.invoke()` -> `llm.ainvoke(messages)`

用途：

- 根据文档、校验规则和已提取 JSON 返回错误数组。

## 9.2 LLMService 当前职责

`LLMService` 当前承担了以下职责：

1. 从配置构建模型实例池
2. 设置 `NO_PROXY`
3. 轮询选择实例
4. 统一执行 `ainvoke`
5. 统一做异常包装
6. 统一做重试和随机退避

## 9.3 当前接入模型方式

当前代码在实现上是 Anthropic 强耦合：

- 直接导入 `langchain_anthropic.ChatAnthropic`
- 实例池类型写死为 `List[ChatAnthropic]`
- `anthropic_api_key` 参数名写死
- 使用了 Anthropic 风格的 `thinking` 参数

这意味着“当前实现并不是通用 Chat Model 工厂”，而是“Anthropic 客户端封装器”。

## 9.4 当前配置模型

`config/config.yaml` 里 `llm_interfaces` 只有以下字段：

- `name`
- `base_url`
- `api_key`
- `model`

这套配置足以表达“多个 Anthropic 兼容端点”，但不足以表达“多供应商差异化参数”。例如：

- OpenAI 常见的是 `api_key` + `base_url` + `model`
- Anthropic 额外可能需要 `thinking`
- OpenAI 新接口可能会涉及 `response_format`
- 未来若接 Azure OpenAI，还可能需要 `api_version`、部署名等字段

换言之，当前配置模型描述的是“一个抽象不足的统一接口”。

## 10. OpenAI 和 Anthropic 接口切换灵活性分析

## 10.1 结论

当前仓库在“提示词层”相对中立，在“调用抽象层”部分中立，但在“客户端实现层”和“配置层”上明显偏向 Anthropic。

可以总结为：

- 提示词兼容性：中高
- 业务流程兼容性：高
- SDK/客户端兼容性：低
- 配置兼容性：低
- 直接切换可行性：低

## 10.2 为什么提示词层相对容易切换

当前对 LLM 的使用方式比较保守：

- 全部是文本提示词
- 没有使用工具调用
- 没有使用函数调用
- 没有使用多模态输入
- 没有依赖 Anthropic 专属输出结构

因此，只要新的模型能稳定遵守“返回纯 JSON”或“返回 CheckError 数组”的格式要求，提示词本身大概率可以继续使用。

## 10.3 为什么实现层不容易直接切换

以下代码特征决定了当前不能只改配置就切到 OpenAI：

### 10.3.1 客户端类写死

`LLMService` 直接使用 `ChatAnthropic`，没有工厂，也没有接口抽象。

### 10.3.2 参数名写死

实例化时使用：

- `anthropic_api_key=...`

这不是 OpenAI 客户端的参数契约。

### 10.3.3 Provider 专属参数写死

实例化时固定传入：

```python
thinking={
    "type": "enabled",
    "budget_tokens": 16000
}
```

这是 Anthropic 风格参数，不适用于 OpenAI 客户端。

### 10.3.4 依赖集不完整

`requirements.txt` 中只有：

- `langchain-anthropic`

并没有：

- `langchain-openai`

因此从依赖层面也还没有准备好切换到 OpenAI。

## 10.4 当前切换弹性评估

### 10.4.1 切到 Anthropic 兼容网关

如果目标仍然接受 Anthropic 客户端协议，只是替换 `base_url` 和 `model`，那么当前代码的切换成本较低。

### 10.4.2 切到 OpenAI 官方或 OpenAI 兼容接口

如果目标要求使用 OpenAI 客户端协议，则当前代码不能无改造直接切换。至少需要：

1. 引入 OpenAI 对应客户端依赖
2. 改造 `LLMService`
3. 在配置中区分 provider
4. 将 provider 专属参数下沉到工厂层

## 10.5 建议的重构方案

建议把 LLM 接入改造成“三层结构”：

### 10.5.1 配置层

为每个接口增加 `provider` 字段，例如：

```yaml
llm_interfaces:
  - name: primary
    provider: anthropic
    base_url: http://...
    api_key: ${ANTHROPIC_API_KEY}
    model: claude-sonnet-4
    options:
      thinking:
        type: enabled
        budget_tokens: 16000
```

也支持：

```yaml
llm_interfaces:
  - name: primary
    provider: openai
    base_url: https://api.openai.com/v1
    api_key: ${OPENAI_API_KEY}
    model: gpt-5-mini
    options:
      reasoning_effort: medium
```

### 10.5.2 工厂层

新增 `llm_factory.py`：

- 根据 `provider` 返回 `ChatAnthropic` 或 `ChatOpenAI`
- 在工厂内部消费 provider 专属参数
- 对外只返回 `BaseChatModel`

### 10.5.3 服务层

`LLMService` 不再感知 `ChatAnthropic`，改为依赖抽象类型，例如 LangChain 的 `BaseChatModel`。

这样：

- 实例池可以混合不同 provider
- 重试逻辑无需关心供应商差异
- 上游节点无需改动

## 11. 可以改进的地方

以下改进项按优先级分组。

## 11.1 P0：会影响正确性的缺陷

### 11.1.1 业务校验在重试达到一定次数后会被强行视为成功

`src/validation_node.py` 中存在：

```python
if is_valid or outer_iteration > 2:
    state = self._handle_validation_success(state)
```

这意味着即便业务校验仍然失败，只要外层迭代次数大于 2，就会把模块标记成成功。这会把错误数据带入最终合并结果。

影响：

- 最终规则可能不满足校验规则
- 下游测试用例生成建立在错误规则之上
- 失败被伪装为成功，影响排障

建议：

- 移除该兜底成功逻辑
- 或改成“进入人工复核状态”
- 或显式输出 `degraded` 状态，不能混同于 `success`

### 11.1.2 测试用例生成器中的标量参数在正常路径下不会被加入输出

`src/test_case_generator.py` 中，标量参数 `input_entry` 的构造和 `case['inputs'].append(input_entry)` 位于异常处理块之后，导致正常情况下不会追加标量参数。

影响：

- 生成的测试用例缺失非 tensor 输入参数
- 生成结果与规则文件不一致

建议：

- 将 `input_entry` 的创建和 `append` 放回 `for` 循环主路径
- 为 tensor 与 attrs 分别增加单元测试

## 11.2 P1：高概率导致行为偏差或维护误判

### 11.2.1 `parameter_constraints` 模块名与数组包装键名不一致

`src/extraction_node.py` 的 `_convert_array_to_dict()` 中使用了：

- `param_constraints`

而系统真实模块名是：

- `parameter_constraints`

影响：

- 当 LLM 返回数组时，包装后的 JSON 键名可能错误
- 与 `common_model_definition.py` 中的目标模型不一致

建议：

- 统一模块名与目标 JSON 字段名
- 用常量表维护，而不是散落在条件分支中

### 11.2.2 配置校验结果未被主流程真正消费

`main.load_and_validate_config()` 调用了 `config_loader.validate_config()`，但没有判断返回值，也没有在失败时中止程序。

影响：

- 明明配置无效，程序仍可能继续运行
- 错误暴露时机会后移到运行中

建议：

- `validate_config()` 失败时抛 `ConfigError`
- 主流程显式终止，而不是仅 `print`

### 11.2.3 提示词规则与模型定义存在漂移风险

例如 `prompts/basic_info_extracting_guide.md` 里描述了：

- `two-step`
- `single-step`

而 `common_model_definition.py` 中 `ApiFlow` 定义为：

- `one-step`
- `two-step`

这说明提示词与代码模型之间已经出现不一致。

影响：

- LLM 可能输出 `single-step`
- Pydantic 校验会拒绝该值

建议：

- 建立“提示词规则与模型 schema 一致性检查”
- 至少对枚举值做自动扫描比对

### 11.2.4 失败迭代次数统计不可信

`src/module_processing_graph.py` 的 `_fail_node()` 用了：

- `state["current_iteration"]`

但该字段并不在单模块状态中递增维护，导致 `total_iterations` 统计不准确。

建议：

- 统一用 `outer_iteration` 和 `inner_iteration` 推导
- 删除无效的 `current_iteration` 遗留逻辑

## 11.3 P2：架构与工程化改进

### 11.3.1 LLM 客户端应做 provider 解耦

当前实现只能稳定支持 Anthropic 风格客户端，建议引入：

- provider 字段
- 工厂模式
- 抽象 chat model 类型

### 11.3.2 配置中不应直接存放明文密钥

`config/config.yaml` 当前直接包含 `api_key`。

建议：

- 改为环境变量引用
- 或配置文件仅保存变量名

### 11.3.3 顶层工作流中手动创建事件循环，复杂度偏高

`src/graph.py` 在同步节点里手工创建 event loop，再执行异步模块处理。该写法能工作，但会增加：

- 可读性成本
- 调试难度
- 与运行环境耦合的风险

建议：

- 尽量使用 LangGraph 原生异步节点
- 避免在已异步的系统中再手工管理子事件循环

### 11.3.4 日志配置未完全遵循配置文件

`setup_logging()` 中实际固定了格式：

- `%(asctime)s - %(message)s`

而不是使用 YAML 中的 `logging.format`。

建议：

- 日志格式统一从配置读取

### 11.3.5 缺少自动化测试

当前仓库依赖 `pytest`，但没有正式测试目录和覆盖核心流程的测试。

建议优先补：

1. `LLMService` 的工厂/重试单元测试
2. `ExtractionNode` 的 JSON 解析测试
3. `JsonValidationNode` 的 schema 校验测试
4. `ValidationNode` 的错误数组解析测试
5. `test_case_generator.py` 的规则到样例转换测试

### 11.3.6 存在遗留或未接线代码

例如：

- `error_merger.py` 基本为空
- `SkillExecutor._backup_rule_file()` 当前未参与主链路
- `LLMService.allocate_modules_to_interfaces()` 与 `get_llm_by_index()` 未进入主链路

建议：

- 删除未使用代码，或补齐接线
- 降低维护噪音

## 12. 推荐演进路线

建议按照以下顺序治理：

### 第一阶段：先修正确性

1. 修正 `validation_node.py` 中的强制成功逻辑
2. 修正 `test_case_generator.py` 的标量参数写入问题
3. 修正 `parameter_constraints` 键名不一致问题
4. 修正提示词与模型枚举不一致问题

### 第二阶段：提升可维护性

1. 清理遗留字段和死代码
2. 改造配置校验为异常式失败
3. 统一日志配置与输出
4. 引入最小测试集

### 第三阶段：重构 LLM 抽象

1. 引入 provider 字段
2. 增加 LLM 工厂
3. 将 `LLMService` 改为依赖抽象模型
4. 接入 `langchain-openai`
5. 为 OpenAI/Anthropic 建立统一调用契约测试

## 13. 总结

这个仓库的核心价值在于：

- 把“算子文档 -> 结构化规则 -> 测试用例”做成了明确流水线
- 已经具备模块化提示词、双层校验和中间结果留存机制
- 整体架构对未来做 provider 解耦是有基础的

当前最主要的问题不在“有没有架构”，而在：

- 正确性保护存在漏洞
- 提示词与模型有漂移
- LLM 接入层对 Anthropic 强耦合
- 自动化测试不足

如果只从“能跑”看，当前系统已经成形；如果从“可长期维护、可切换模型供应商、可稳定产出正确结果”看，仍需要一轮比较系统的治理。
