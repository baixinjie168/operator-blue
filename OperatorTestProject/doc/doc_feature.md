# 当前项目未实现能力清单

## 说明

本文基于 [doc_v3.md](./doc_v3.md) 的 V3 规划，结合当前仓库代码现状整理。

判断口径如下：

- 以仓库内已有、可重复执行的代码能力为准。
- 不将人工临时整理 `workspace_2`、批量改 JSON、手工复制文件等操作视为“项目已实现能力”。
- 当前项目主链路可以概括为：本地文档输入 -> 模块抽取 -> JSON 结构校验 -> 业务校验 -> 规则合并 -> 测试用例生成。

## 当前项目已具备的基线能力

为避免将已实现能力误判为缺失，这里先明确当前项目已有的能力边界：

- 支持读取本地算子 Markdown 文档并执行抽取主流程。
- 支持按模块抽取结构化信息，包括 `parameter_constraints`、`basic_info`、`inter_parameter_constraints`、`platform_specifics`、`other_parameters`、`functions`、`dtype_map`。
- 支持 JSON 结构校验与业务校验。
- 支持将模块结果合并为 `*_extracted_constraints.json`。
- 支持基于规则文件生成测试用例。
- 支持基于已有 constraint JSON 手动生成 case。

## 对照 V3 规划，当前未实现的能力

### 1. MCP 文档资产服务未实现

`doc_v3.md` 中将 `MCP 文档资产服务` 作为 V3 核心增强能力之一，用于：

- 历史文档查询
- 历史规则复用
- 文档版本差异比对

当前仓库主入口仍然是直接读取本地 Markdown 文件，不存在 MCP 资产服务层，也没有资产查询、索引、检索或统一访问接口。

### 2. 历史规则自动复用能力未实现

V3 规划要求：

- 对于已处理且文档未变更的算子，直接复用历史 `*_extracted_constraints.json`
- 不再重复进行全量抽取

当前项目虽然支持“拿已有 constraint JSON 继续生成 case”，但这是手动指定文件路径的能力，不是自动识别“已有历史资产且可复用”的业务链路。

也就是说，当前缺少：

- 历史资产存在性判断
- 文档是否变更的判断
- 自动切换到“复用路径”的调度逻辑

### 3. 文档版本差异比对能力未实现

V3 规划要求：

- 查询历史原文档
- 执行新旧文档差异比对
- 输出版本变更结论

当前仓库中的版本目录主要用于抽取迭代过程中的中间结果保存，不是“文档历史版本管理”。

因此当前缺少：

- 历史原文档存档与索引
- 新旧文档 diff 分析
- 面向规则提取的差异摘要生成

### 4. 增量约束发现与增量提取路径未实现

V3 规划要求：

- 文档变更后，不是完全重跑全量提取
- 而是根据差异结果生成“需要补充哪些新增约束”的增量提示

当前项目只有固定的全量抽取主路径，没有：

- 增量提示生成器
- 增量规则抽取模式
- 全量与增量结果的合并策略

### 5. 三种执行模式未实现完整闭环

`doc_v3.md` 中定义了三种执行模式：

- 全量提取
- 历史规则复用
- 文档变更后的增量提取

当前项目实际落地的只有“全量提取 + 合并 + 用例生成”这条单一路径，缺少另外两条自动化执行模式。

### 6. Review UI / 审批层未实现

V3 规划中存在：

- Review UI
- 审批层
- 审批后形成正式规则版本

当前仓库没有前端页面、服务端接口、任务审批流、人工 review 交互界面，也没有“待审批/已审批/正式发布”这类状态管理机制。

当前只具备：

- 错误文件保存
- 中间版本文件保存

这些能力不能等价视为 Review UI 或审批流程。

### 7. 正式规则版本管理与发布机制未实现

V3 规划强调：

- 审批后形成正式规则版本
- 正式规则进入后续测试和复用链路

当前项目虽然会保存抽取结果和中间版本，但缺少：

- 正式版本标识
- 版本发布动作
- 已发布规则和草稿规则的区分
- 规则版本生命周期管理

### 8. 发布到 ATK 的执行链路未实现

当前项目中出现了若干“适配 ATK 数据格式”的字段和注释，但这不等于已经实现了 ATK 集成。

当前缺少：

- 将生成结果发布到 ATK 的接口
- ATK 任务创建/触发机制
- 任务执行状态跟踪
- 执行完成后的结果拉取

因此，现状更准确地说是“面向 ATK 的数据结构适配准备”，而不是“已接入 ATK 测试平台”。

### 9. ATK 结果分析引擎未实现

V3 规划要求测试执行后具备：

- ATK 结果分析
- 错误归因
- 经验提炼

当前仓库没有看到：

- ATK 结果解析模块
- 失败模式统计
- 自动生成经验结论的逻辑
- 将结果反哺到抽取/生成流程中的机制

### 10. 覆盖率执行与覆盖率分析能力未实现

V3 规划要求：

- 覆盖率工具执行
- 覆盖率分析
- 覆盖率经验沉淀

当前仓库未发现覆盖率执行器、覆盖率报告解析器、覆盖率统计汇总或覆盖率反馈模块。

### 11. 知识资产层未实现

V3 架构中的知识资产层包含：

- Prompt / Template Registry
- Constraint DSL Registry
- Review Memory
- ATK 结果经验库
- Coverage 经验库

当前项目虽然有 `prompts/` 目录和若干规则文件，但仍属于静态文件集合，并未形成真正的“知识资产层”。

当前缺少：

- 统一注册中心
- 元数据管理
- 可检索能力
- 版本管理能力
- 回灌更新能力

### 12. 测试反馈知识回灌闭环未实现

V3 规划要求将：

- ATK 测试结果
- 覆盖率结果

统一进入经验提炼链路，再反向优化：

- 提示词
- 模板
- 约束表达语言

当前项目没有形成这条闭环，测试执行结果不会自动进入后续抽取策略优化。

### 13. 知识资产管理模块未实现

V3 业务架构中特别强调了“知识资产管理模块”作为反馈沉淀的承接点。

当前仓库没有独立的知识资产管理模块，缺少：

- 经验条目录入
- 经验检索
- 经验与算子的关联关系管理
- 将经验用于下一轮抽取的可编排机制

## 有一定痕迹，但尚未真正落地的点

这部分不应算作“已实现”，但说明仓库中存在一定预留或雏形。

### 1. 错误版本衔接能力只有部分预留

仓库中存在：

- `previous_error_path`
- 错误 JSON / Markdown 合并

但当前更多是为了保存和串联校验错误，不是 V3 所说的“历史知识复用”和“反馈回灌”。

### 2. 规则备份目录存在，但不等于资产服务

仓库中存在 `backup_dir` 配置和规则备份逻辑，但目前更像简单文件备份，并没有形成：

- 历史文档检索
- 版本差异分析
- 资产自动复用

### 3. ATK 相关字段适配存在，但不等于平台集成

仓库中已有面向 ATK 的类型映射和 case 结构适配，但没有真正的：

- ATK 发布
- ATK 执行
- ATK 结果回收
- ATK 结果分析

## 总结

如果按 `doc_v3.md` 的目标来评价，当前项目更接近：

- `V2/V2.5` 阶段的“规则抽取 + 校验 + 合并 + 用例生成工具链”

而不是完整的：

- “可复用、可增量、可反馈进化”的 V3 测试生产平台

当前最核心的缺口集中在四个方向：

- `MCP 资产复用`
- `文档版本差异与增量提取`
- `ATK/覆盖率执行结果分析`
- `知识资产回灌闭环`
